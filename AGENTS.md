# AGENTS.md - LiteracyBee Platform

## Project Overview
**LiteracyBee** is a full-stack literacy education platform with role-based access (parent, child, admin). The backend uses Node.js/Express with Prisma ORM, and the frontend uses React with Redux for state management.

```
Back-End/          → Express API server (port 3000)
literacybee-frontend/ → React app (localhost:3000)
PostgreSQL         → Dockerized database
```

---

## Critical Architecture Patterns

### 1. **Three-Layer Backend Architecture**
- **Controllers** (`Back-End/controllers/`) - HTTP request handling, input validation, error mapping
- **Services** (`Back-End/services/`) - Business logic, reusable operations (e.g., `AuthService.generateToken()`)
- **Models** (`Back-End/models/`) - Database operations via Prisma

**Example Flow:**
```
Request → Controller validates input → Service handles logic → Model queries DB → Response
```

### 2. **Role-Based Access Control**
Three user roles with specific authorization:
- **child**: Can only access their own lessons/progress
- **parent**: Can view/manage their children's data
- **admin**: Full system access

**Key middleware:** `authMiddleware.js` - `authenticate()` extracts JWT; `requireRole()` checks role; `authorizeChildAccess()` handles parent-child authorization.

**Important:** Parent access is verified by matching `child.parent_id === req.userId` (line 56 in authMiddleware.js).

### 3. **Authentication Flow**
- Parents/admins: JWT via email+password (hashed with bcrypt)
- Children: JWT via parent ID + PIN (no email)
- Frontend stores token in `localStorage`, passed via `Authorization: Bearer <token>` header
- `authMiddleware.js` validates JWT and populates `req.userId`, `req.userRole`, `req.user`

**Error Convention:** Services throw named errors like `'EMAIL_EXISTS'` or `'INVALID_CREDENTIALS'` for controllers to map to HTTP status codes.

## Data Model (Prisma Schema)

### Core Entities
- **users**: Hierarchy with parent-child relationship via `parent_id`
- **units** → **lessons** → **exercises**: Learning content hierarchy
- **exercise_attempts**: Track individual exercise responses with XP earned
- **progress**: Child's lesson completion status (unique constraint: one per child per lesson)
- **streaks, badges, user_badges**: Gamification system

**UUID primary keys generated by PostgreSQL** (`gen_random_uuid()`), datetime fields use `Timestamptz(6)`.

### Key Relationships
```
users (parent) --< users (children via parent_id)
units --< lessons --< exercises
users --< exercise_attempts --> exercises
users --< progress --> lessons
users --< streaks (1:1)
users --< user_badges --> badges
```

## Frontend Architecture

### Redux Store Structure
```javascript
// store.js: Three slices
auth: { user, token, isAuthenticated }
child: { selectedChild, children }
lesson: { lessons, currentLesson }
```

### API Integration
- **api.js**: Axios instance with auto-injection of JWT token in all requests
- Base URL from `REACT_APP_API_URL` env var (defaults to `http://localhost:3000/api`)
- Request interceptor adds `Authorization: Bearer <token>` if token exists in localStorage

### Route Protection
- **PrivateRoute.jsx**: Wraps routes, checks Redux auth state + allowed roles
- Only authenticated users with correct role can access protected pages
- Example: `/parent/dashboard` only accessible if `auth.user.role === 'parent'`

## API Structure

### Endpoints by Domain
```
POST   /api/auth/register       → registerParent (AuthService)
POST   /api/auth/login          → loginParent
POST   /api/auth/create-child   → createChild (requires parent JWT)
POST   /api/auth/child-login    → loginChild

GET    /api/lessons             → List all published lessons
GET    /api/lessons/:id         → Get lesson with exercises
POST   /api/lessons/:lessonId/start → Start lesson (track progress)

GET    /api/progress/:childId   → Child's progress across lessons
POST   /api/progress/:childId/submit → Submit exercise attempt

GET    /api/admin/users         → List all users (admin only)
POST   /api/admin/badges        → Create badge (admin only)
```

**Response Convention:** Errors return `{ error: 'user-friendly message' }` with appropriate HTTP status (400, 401, 403, 409, 500).

## Development Workflows

### Local Setup
```bash
# Backend
cd Back-End
npm install
docker-compose up -d          # Starts PostgreSQL on port 5432
npx prisma migrate dev        # Run migrations
node index.js                 # Start API on port 3000

# Frontend
cd literacybee-frontend
npm install
```

- Run `npx prisma migrate dev --name <feature>` to create migrations
- Migrations live in `Back-End/prisma/migrations/`

## Project-Specific Conventions

1. **Error Handling in Services**: Throw custom error messages (no stack traces) for controllers to catch. Controllers map these to HTTP status codes:
   ```javascript
   // Service
   if (existing) throw new Error('EMAIL_EXISTS');
   
   // Controller
   catch (err) {
     if (err.message === 'EMAIL_EXISTS') return res.status(409).json({ error: 'Email already exists' });
   }
   ```

2. **Security Patterns**:
   - JWT stored in localStorage (frontend) or memory (optional)
   - Passwords hashed with bcrypt (BCRYPT_ROUNDS env var configures iterations)
   - Rate limiting: 100 req/15min on `/api/auth`, 500 req/15min on others
   - Helmet for HTTP headers, CORS enabled

3. **Child Access Authorization**: 
   - Children can only access their own data (line 46 in authMiddleware.js)
   - Parents verified by their relationship (line 56)
   - Always use `authorizeChildAccess(childId)` middleware on child-specific routes

4. **Gamification Points**:
   - Lessons reward XP on completion (`xp_reward` field, default 50)
   - Exercises reward XP on correct answer (`xp_value` field, default 10)
   - Badges tied to criteria: `lessons_completed`, `total_xp`, `streak_days`

5. **Frontend Redux Pattern**:
   - Dispatch thunks or actions from pages
   - Selectors retrieve state
   - API calls in thunks; dispatch actions with results
   - PrivateRoute always checks auth + role before rendering

## Common Tasks for Agents

### Adding a New Feature
1. Update Prisma schema if new data entity needed
2. Run migration: `npx prisma migrate dev --name feature_name`
3. Create model methods in `Back-End/models/`
4. Add service logic in `Back-End/services/`
5. Create controller in `Back-End/controllers/`
6. Mount routes in appropriate route file
7. Apply `authenticate` and `requireRole` middleware
8. Test via Postman or curl with Authorization header

### Adding Admin-Only Endpoint
- Use `requireRole('admin')` middleware
- Example: `router.post('/badges', authenticate, requireRole('admin'), adminController.createBadge)`

### Modifying Child Access Rules
- Update `authorizeChildAccess()` logic in `authMiddleware.js`
- Remember: role==='child' checks `req.userId === childId`, role==='parent' checks parent relationship
